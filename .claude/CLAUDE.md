## Project Overview

GraphSense Dashboard — a web GUI for interactive cryptocurrency analysis, written in **Elm 0.19.1** and built with **Vite**. Uses The Elm Architecture (TEA) pattern: Model → Msg → Update → Effect.

## Common Commands

```bash
# Development
make serve              # Start dev server (localhost:3000) — runs prepare + codegen + vite
make build              # Production build to ./dist
make test               # Run Elm tests (elm-test-rs)
make format             # Format Elm code (elm-format)
make lint               # Lint with elm-review
make lint-fix           # Auto-fix lint issues

# Setup (first time)
npm install
make install            # Install pre-commit hooks
cp config/Config.elm.tmp config/Config.elm
cp env.template .env

# Code generation
make clean-generated-plugins && make   # Regenerate plugin code after plugin changes
make theme-refresh                      # Re-fetch Figma theme and regenerate

# Docker
make build-docker       # Build Docker image
make serve-docker       # Run in Docker
```

Single test files: not supported by elm-test-rs natively; tests live in `tests/`.

## Architecture

### Core TEA Flow (src/)

- **Main.elm** — `Browser.application` entry point
- **Model.elm** — Central state type (`Model`) and top-level `Msg` type
- **Update.elm** — Main update function dispatching to domain-specific updaters
- **View.elm** — Top-level view routing to page views
- **Effect.elm** — Side-effect abstraction wrapping `Cmd msg` with batching
- **Sub.elm** — Subscriptions (window resize, time, keyboard, etc.)
- **Route.elm** — URL routing; routes: `Graph`, `Pathfinder`, `Home`, `Stats`, `Settings`, `Plugin`
- **Ports.elm** — JavaScript interop (localStorage, clipboard, canvas, file export)
- **main.js** — JS bootstrap, port wiring, external library initialization

### Directory Layout Under src/

- `Model/` — Domain-specific data types (Graph, Pathfinder, Search, etc.)
- `Msg/` — Message types per domain
- `Update/` — Domain-specific update logic
- `Effect/` — Effect handlers (API calls, ports, navigation)
- `View/` — Page-level views (Graph, Pathfinder, Search, Stats, etc.)
- `Components/` — Reusable UI components
- `Css/` — Elm-CSS styles
- `Theme/` — Theme definitions
- `Util/` — Helper modules
- `Api/` — Generated REST API client (from OpenAPI)
- `PluginInterface/` — Plugin hook definitions

### Code Generation Pipeline

Several directories are **generated** — do not edit directly:
- `generated/plugins/Plugin.elm` — generated from `plugin_templates/*.mustache` + `config/Config.elm`
- `generated/theme/` — generated from Figma via `tools/codegen.sh`
- `generated/utils/RecordSetter.elm` — generated by `setem` from model types
- `elm.json` — generated from `elm.json.base` (edit the base file instead)

The generation script `tools/generate.js` reads active plugins from `config/Config.elm`, processes mustache templates, and produces the plugin glue code.

### UI Components and Styling (Figma-generated)

UI elements are generated from Figma designs. Run `make theme-refresh` to fetch new definitions. **Always prefer using the generated Figma components over writing direct CSS/HTML.** Never manually edit any files under `generated/` — they will be overwritten by the next codegen run.

Key rules:
- **Colors**: All colors live in the generated `Colors.elm` (light) and `ColorsDark.elm` (dark mode) under `generated/theme/Theme/`. Never introduce ad-hoc color values; always reuse colors from there.
- **Icons**: SVG icons live in `Theme.Html.Icons` and `Theme.Svg.Icons` (under `generated/theme/Theme/Html/Icons.elm` and `Svg/Icons.elm`). Each icon has an html variant and an svg-only variant.
- **Component variants**: Every generated Figma component has three variants:
  - **Plain** — static, no customization
  - **WithAttributes** — same interface but allows redefining attributes on specific nodes (e.g. adding dynamic hover effects)
  - **WithInstances** — same interface as WithAttributes but allows replacing entire sections/sub-components (for advanced styling and dynamic content)

### Plugin System

Plugins live in `plugins/` (often symlinks to external repos). Each plugin can have its own `elm.json`, `package.json`, `lang/`, and `public/` directories. Active plugins are declared in `config/Config.elm` by importing and registering them via `Plugin.empty |> Plugin.myplugin (...)`.

Plugins hook into the app through `src/PluginInterface/` interfaces (View, Update, Effects, Routes).

### API Layer

- OpenAPI-generated client in `openapi/src/Api.elm` (regenerate with `make openapi`)
- `Effect.Api` orchestrates HTTP requests with auth headers
- `RemoteData` pattern (`NotAsked | Loading | Success | Failure`) for async state

### i18n

Translation files in `lang/*.yaml` (en, de, es, it). Plugin translations in `plugins/*/lang/`. Synchronized with `tools/harmonize_lang_files.py`.

## Testing

The test suite is small but growing. When adding or changing logic — especially pure functions, data transformations, and parsing — write tests for it. Easy-to-test logic should always be tested. Run tests with `make test`.

## Key Configuration

- `config/Config.elm` — Plugin registration, theme, logout URL (not committed; copy from `.tmp`)
- `.env` — `VITE_GS_REST_URL` (required), optional service endpoints
- `vite.config.mjs` — Dev server on port 3000, public dir is `generated/public`
- Elm packages cached in `elm_packages/` (via `ELM_HOME` override)

## Pre-commit Hooks

Runs on commit: `make format`, `make lint`, `make test`. On push: `tools/set_version.sh` (writes version to `Version.elm`).

## InfiniteTable (Virtual Scrolling)

`Components.InfiniteTable` provides virtual scrolling for large datasets. **It requires fixed/constant row heights.** The scroller estimates total content height as `rowHeight * itemCount` using a single sampled row. Variable-height rows cause spacer miscalculations, scrollbar jitter, and can prevent data loading at scroll boundaries. When converting a table to use `InfiniteTable`, always ensure rows have a fixed CSS height (no `height: auto`, no text wrapping that grows rows). If variable row content is needed, truncate with `text-overflow: ellipsis` and use tooltips for the full text.

## Patched Dependencies

The Makefile clones patched forks of `elm/virtual-dom`, `elm/browser`, `elm/html`, and `rtfeldman/elm-css` into `elm_packages/` (the `virtual-dom-fix` target). These patches prevent the app from crashing when browser extensions modify the DOM, which would otherwise conflict with Elm's virtual DOM diffing. They are required for the build to work.
